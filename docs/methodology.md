# MESSINA Methodology Note

## Version Scope

This public repository is centered on **v25.9**, the current public-facing baseline of the framework.
It provides a bounded, publication-safe comparison surface rather than a full release of the historical internal engine.

This note is aligned to the current public V25.9 line while preserving the methodological discipline established across the V25.1-V25.9 publication-safe trajectory. It should remain stable unless the public line materially changes its aggregation, validation, or observer-diagnostics claims.

## Scope and Design Aim

MESSINA is a mechanism-encoding, research-oriented scenario-exploration framework designed to compare structured geopolitical crisis worlds under controlled assumptions. It is not a direct forecasting tool and should not be read as a production-grade decision engine or an empirical probability model.

The framework is built to preserve key causal and quasi-causal relationships inside a synthetic scenario environment so that comparative stress patterns, propagation paths, and governance tradeoffs can be examined under variation.

The safest general description is:

> MESSINA is a mechanism-encoding, research-grade scenario-exploration framework whose outputs should be read as branch-dominance signals under encoded assumptions.

## Engine and Case Separation

A central rule of MESSINA is that the engine and the case should not be confused.

- The **engine** contains the general crisis logic, layer interactions, propagation structure, aggregation machinery, and validation surfaces.
- The **case** contains scenario-specific assumptions, chronology encoding, overlays, events, actor settings, and provenance.

This distinction is essential. Without it, the framework collapses into a single narrative artifact and becomes harder to audit, extend, or compare.

## Architecture / Analytics / Protocol

MESSINA is organized around three linked components.

### Architecture
Architecture builds structured crisis worlds under encoded assumptions.

### Analytics
Analytics reads repeated runs and comparative differences. In the public line, this includes branch dominance, scenario comparison, governance-diagnostic interpretation, and aggregation-aware reading.

### Protocol
Protocol translates analytical readings into policy-facing crisis-navigation implications. It is not deterministic command logic.

A case layer sits alongside these components in the form of the scenario pack.

## Layered and Mixed-Resolution Structure

MESSINA uses a layered mixed-resolution structure because crisis dynamics do not remain legible at one scale.

The current public line should describe these layers explicitly:
- **Nano** for discrete triggers, signal injections, and bounded local shock elements;
- **Micro** for lower-level heterogeneity, behavior distributions, and early burden asymmetries;
- **Mezzo** for clustered institutional or sectoral aggregates, including island-like semi-cohesive actor groupings;
- **Network** for lateral propagation across coupled channels and connected nodes;
- **Macro** for broader phase, plateau, regime, and compressed risk states;
- **Module** for functional bundles such as governance diagnostics, corridor logic, strategic-buyer logic, and credibility handling.

A key clarification follows from this: **modules are not islands**. Islands are Mezzo-level aggregates. Modules are functional mechanism bundles that may read or process signals from multiple scales.

## Dynamics and Output Philosophy

The framework is not designed to answer "what will happen?" in a narrow deterministic sense. It is designed to answer questions such as:
- what kinds of downstream patterns become more likely under one governance setting than another,
- which pathways dominate under specific assumptions,
- how much crisis structure is hidden by terminal averages,
- which dimensions of control deteriorate before visible collapse occurs.

Outputs should therefore be read as comparative structured signals rather than literal forecasts.

## Provenance Discipline

Scenario inputs should carry explicit provenance wherever possible. The point of provenance is not to imply perfect calibration. It is to make the source, rationale, confidence, and evidentiary status of assumptions inspectable.

A good provenance discipline should distinguish at least:
- source type,
- source reference,
- rationale,
- confidence,
- whether a value is calibrated, estimated, stylized, or placeholder.

## Monte Carlo, Replay, and Stochastic Discipline

The broader MESSINA lineage includes bounded Monte Carlo use rather than unlimited stochastic proliferation.

Historically, the most important public-safe reference point is **MC512**.

MC512 should be defended narrowly:
- useful for branch stability,
- useful for phase consistency,
- useful for scenario comparison,
- useful for publication-level stress-testing under encoded assumptions,
- **not** sufficient for strong rare-event frequency precision claims.

This distinction matters. Repetition alone does not convert model outputs into real-world probabilities.

The public methodological line should therefore say three things clearly:
1. repeated runs are used to stabilize comparative interpretation;
2. replay and seed discipline matter;
3. Monte Carlo repetition does **not** justify strong probability claims about the external world.

## Current Aggregation Position

Aggregation has evolved substantially across the V25 line. Earlier public notes accurately described aggregation as largely weighted/composite, with only limited variance-aware behavior in selected stages. That description is no longer sufficient for the current architecture discussion.

The current methodology position should be stated more precisely.

### What is already present

The public V25 direction now includes:
- summary aggregation,
- path aggregation,
- regime aggregation,
- dispersion-aware comparison,
- scenario comparison,
- sensitivity reporting,
- bootstrap-oriented uncertainty handling in limited form.

### What this means

Aggregation is no longer just a terminal summary layer. It increasingly functions as the main interpretive bridge between raw simulation dynamics and validation outputs.

### What is still missing

Even so, the framework has not yet fully matured into a research-grade hierarchical aggregation stack. The following remain partial, proposed, or version-dependent:
- uncertainty decomposition by source,
- event-chain and lead-lag aggregation,
- robust cross-pack discrimination,
- explicit critical-transition monitoring,
- formal cross-scale similarity diagnostics,
- first-class fractal signature outputs.

## Public-Safe Reproducibility Boundary

The current public reproducibility surface is deliberately narrower than the full historical internal engine.

The public-safe runner is designed to reproduce:
- documented governance-observer formulas,
- single-run governance signals,
- same-shock / different-governance comparison outputs,
- example reports and comparison summaries.

It should **not** be described as a full release of the historical internal codebase, a complete multi-version architecture history, or a production deployment surface.

This boundary matters because it keeps the public build aligned with what can actually be inspected and rerun.

## Validation Philosophy

Validation in MESSINA is internal and comparative before it is empirical.

That means the first questions are:
- is the model internally stable,
- are transitions interpretable,
- are differences between scenarios reproducible,
- do summary, path, and regime outputs cohere,
- do sensitivity results make sense relative to the model's own architecture?

Empirical validation, where later attempted, should be layered on top of this rather than assumed from the beginning.

## Path and Regime Aggregation

Current methodology should explicitly recognize the importance of path and regime analysis.

- **Path aggregation** asks how runs arrive at outcomes over time.
- **Regime aggregation** groups runs into behaviorally meaningful classes.

This matters because similar terminal metrics can conceal very different trajectories. A late lock-in path and an early fragmented climb may produce similar peak scores while implying different strategic interpretations.

## Fractal and Complex Systems Clarification

MESSINA already qualifies as multiscale and complex-systems aware. It contains strong cascade logic, heterogeneity, path dependence, threshold behavior, and cross-domain propagation.

It should **not** yet be described as a formal fractal-analytics system.

The methodology line should state this directly:
- cross-scale recurrence is part of the design intuition,
- explicit fractal diagnostics are still proposed observer-layer extensions,
- current outputs do not yet amount to full measured fractal analysis.

A disciplined next step is to add observer-level diagnostics without mutating engine behavior in the first cut.

## Cross-Theater Reuse Boundary

The current public case surface is Hormuz-centered. Historically, the broader design lineage also considered other chokepoint and corridor logics, including Bab el-Mandeb and other cross-theater comparison ambitions.

That does **not** mean the current public outputs should be transferred mechanically across theaters.

A good public rule is simple:
- use Hormuz-centered outputs to reason comparatively about chokepoint structure,
- do not treat them as plug-and-play results for Bab el-Mandeb, Suez, Panama, Taiwan / SCS, or other theaters with different legal, logistical, alliance, and commodity structures.

## Epistemic Boundaries

The framework does not justify empirical causal claims simply because it produces structured outputs.

It does not justify strong real-world probability claims from Monte Carlo repetition alone.

It does not justify turning internal composite scores into public truth claims without careful interpretation.

## Current Best Description

The most accurate current methodological description is this:

MESSINA is a multiscale, mixed-resolution, mechanism-encoding scenario-exploration framework with an increasingly sophisticated aggregation layer. It is already complex-systems aware. It is not yet a full fractal-analytics system. Its most promising frontier lies in making cross-scale structure, critical transitions, and aggregation-aware validation more explicit without compromising engine stability or overstating empirical certainty.

## Worked Public Example: Governance Erosion

The public governance erosion diagnostic is fully visible in `src/public_safe/observer_formulas.py` and `analytics/formula_sheet.md`.

It combines four public signals:

```text
governance_erosion =
    0.30 * (1.0 - epistemic_integrity)
  + 0.25 * (1.0 - decision_friction)
  + 0.25 * platform_dependency
  + 0.20 * mosaic_response
```

This is a design-weighted diagnostic. It is not an empirically estimated coefficient model. Its value lies in making a structured claim inspectable: when institutional friction falls and platform dependency rises, governability can deteriorate even if peak stress barely changes.
